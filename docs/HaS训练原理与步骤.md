# 第零章：最底层的问题——"模型"到底是什么？

先建立一个最重要的心智模型：

> **一个大语言模型，本质上就是一大堆数字（叫"参数"或"权重"），加上一套用这些数字做乘加运算的程序。**

所谓 **Qwen3-0.6B**，"0.6B" = 0.6 Billion = **约 6 亿个数字**。这些数字分成一层一层的矩阵（比如一个 1024×1024 的矩阵就有 104 万个数字）。这个模型有 28 层（`qwen3.block_count = 28`），每层若干矩阵，加起来约 6 亿个参数。

**它怎么"读文章"和"写文章"？**

1. **分词（tokenize）**：先把文本切成 token（可以粗略理解为"字/词碎片"）。例如 `"CURP HEMJ850312"` 可能被切成 `["CUR", "P", " HE", "MJ", "85", "03", "12"]` 这样的碎片，每个碎片对应一个词表里的编号（Qwen3 词表有 151,936 个 token）。
2. **查表 + 计算**：每个 token 编号先查表变成一个 1024 维的向量（一串 1024 个数字），然后依次穿过 28 层矩阵运算，每层都是"向量 × 矩阵 + 非线性变换"。
3. **预测下一个 token**：最后一层输出一个 151,936 维的向量，表示"下一个 token 是词表中每一个的概率"。取概率最大的（或按概率抽样），就生成一个 token。把这个 token 接在原文后面，重复这个过程，就一个一个字地"写"出了回答。

**所以"模型很笨"**：它不会思考，只是在做超大规模的乘加运算。**"模型又很聪明"**：因为那 6 亿个数字是"训练"出来的，训练让它们编码了语言规律和知识。

## 0.1 模型文件：这些数字存在哪？

打开你下载的 `models/HaS_Text_0209_0.6B/` 会看到：

```
model.safetensors   ← 1.2GB，就是那 6 亿个数字本身（权重）
config.json         ← 模型的"结构图纸"：28层、每层维度1024、词表151936...
tokenizer.json      ← 分词器：token <-> 编号的对照表和切分规则
chat_template.jinja ← 对话模板：把"用户说/助手说"拼成模型见过的文本格式
```

`safetensors` 只是一种"存数字的文件格式"（类似 `.npy`），HuggingFace 生态标准。

**"架构"（architecture）= 图纸，"权重"（weights）= 图纸下造出来的具体东西。**
同一个 Qwen3 图纸，灌入不同的权重，就是不同的模型。`config.json` 里 `"architectures": ["Qwen3ForCausalLM"]` 说明 HaS 用的图纸和 Qwen3 **一模一样**，差别只在权重数字不同——这一点后面很关键。

# 第一章：训练是什么？——梯度、损失函数、优化器

## 1.1 训练的朴素思想：做题、对答案、改错

想象教小孩做题：

1. 给一道题（输入），小孩写出答案（模型的预测）；
2. 和标准答案对比，看差多少（**损失函数** loss）；
3. 小孩调整脑回路，下次争取错得少一点（**更新参数**）。

重复几万次，小孩就"学会了"。训练神经网络一模一样，只是"调整脑回路"有精确的数学方法。

## 1.2 损失函数：把"错多少"变成一个数字

模型逐 token 生成文本。对每一个位置，模型输出的是词表上 151,936 个 token 的概率分布，而标准答案是确定的某个 token。

**交叉熵损失**（cross-entropy loss）的直觉：

```
loss = -log(模型分给正确答案的概率)
```

- 模型对正确答案是 90% 把握 → loss = -log(0.9) ≈ 0.1，很小，夸它；
- 模型只有 1% 把握 → loss = -log(0.01) ≈ 4.6，很大，罚它。

训练目标：**让所有训练样本的平均 loss 尽量小**。loss 是所有 6 亿个参数的函数——改变任何一个参数，loss 都会变一点点。

## 1.3 梯度：该往哪个方向改、改多少？

问题来了：6 亿个参数，每个该调大还是调小、调多少？

这就是**微积分**登场的地地方。**梯度（gradient）** = loss 对每个参数的偏导数，含义是"这个参数调大一点点，loss 会变大还是变小、变化多快"。

一个最小例子（这就是你 `微积分入门复习指南.md` 里的内容在发光）：

```
假设模型是 y = w·x，当前 w=3，训练样本是 x=2, 正确答案 y=10
预测: ŷ = 3×2 = 6，loss = (6-10)² = 16
梯度: d(loss)/dw = 2(wx-y)·x = 2×(6-10)×2 = -16
```

梯度是 **-16**（负数），意思是"w 调大，loss 会变小"。于是：

```
新 w = 旧 w - 学习率 × 梯度 = 3 - 0.01×(-16) = 3.16
新预测 = 3.16×2 = 6.32，比 6 更接近 10 了 ✓
```

这个"**沿着梯度的反方向走一小步**"的方法就是**梯度下降（gradient descent）**。

对 6 亿个参数的模型，**反向传播（backpropagation）**算法用链式法则高效算出**所有参数**的梯度——这就是 PyTorch 里 `loss.backward()` 一行代码背后做的事。

## 1.4 优化器：真正执行"改参数"的组件

**优化器（optimizer）**就是根据梯度更新参数的规则。

- **SGD**（随机梯度下降）：最朴素，`w = w - lr × 梯度`，其中 `lr`（学习率，learning rate）是步长，比如 1e-4 = 0.0001。学习率是训练最重要的超参数：太大容易"走过头"（loss 震荡甚至爆炸），太小训不动。
- **Adam**：工业界默认选择。它给每个参数单独维护两个滑动统计量（梯度的均值 m、梯度的平方均值 v），自动调节每个参数的实际步长——陡坡走小步、平地走大步。

**Adam 的代价（这解释了为什么全量训练吃内存）**：

```
每个参数需要额外存 2 个统计量 (m 和 v)，通常用 fp32（4 字节）存储
0.6B 参数 → 仅优化器状态就要 0.6B × 2 × 4字节 ≈ 4.8 GB
再加上：权重本身 + 梯度 + fp32 主权重副本 ≈ 10 GB 起步
```

这就是为什么在你的 24GB Mac 上，我们选择 **LoRA**（只训练 0.5% 的参数，优化器状态从 4.8GB 缩到 ~25MB），第三章详述。

## 1.5 预训练 vs 微调：两种"训练"，成本天差地别

| | 预训练（pretraining） | 微调（fine-tuning） |
|---|---|---|
| 干什么 | 从**随机数字**开始，读几万亿 token 的互联网文本，学会语言和世界知识 | 在训好的模型上，用几千~几万条"问答对"，学会**特定任务/格式** |
| 类比 | 让小孩读完整个图书馆，成为"有文化的人" | 教一个有文化的人"填表"这种具体技能 |
| 数据量 | 数万亿 token | 几千条样本就够（本项目 2000 条） |
| 算力 | 数百张顶级 GPU 跑几周，花费百万美元级 | 一台 Mac 跑几十分钟 |
| 结果 | **基座模型（base model）** | **微调模型（fine-tuned model）** |

**关键认知：微调不改变模型"懂什么"，主要改变模型"怎么表现"**——把已有的知识，用你要求的格式和规则输出出来。我们要教 CURP，本质是教模型"看到这种 18 位编码 + 这种问法时，把它归到这类 type 下"，模型读西班牙语文本的能力早就在预训练时学好了。

# 第二章：HaS 模型到底是什么？——基座、微调、协议、量化

## 2.1 HaS 不是从零研发的模型——而这很正常、很标准

直接看证据。ModelScope 官方仓库的元数据写着：

```json
"Architectures": ["Qwen3ForCausalLM"],
"BaseModel": ["Qwen/Qwen3-0.6B-Base"],
"BaseModelRelation": "finetune"
```

翻译过来：**HaS 文本模型 = 阿里巴巴开源的 Qwen3-0.6B-Base（基座） + 腾讯玄武实验室的监督微调（SFT）**。

**"基座模型（base model）"是什么？** 就是 1.5 节说的预训练产物——读完了海量文本、懂 8 国语言、但**没有接受过任何"按指令办事"的训练**的原始模型。你问它问题，它可能会续写一段类似问题的文本而不是回答你，因为它只学过"文本后面通常接什么"，没学过"问题后面应该接答案"。

**那玄武实验室的贡献是什么？** 绝不只是"微调了一下"这么简单：

1. **数据工程**（最核心）：构造了覆盖 **约 70,000 种实体类型**、8 种语言的高质量 SFT 训练数据。业界共识：微调的效果 80% 取决于数据质量，这是真正的护城河；
2. **任务格式设计**：发明了 `<Type[ID].Category.Attribute>` 三级语义标签体系，让脱敏后的文本仍能被云端 LLM 理解；
3. **系统工程**：把 NER / Hide / Restore / Pair 拆成 6 个模型任务 + 6 个确定性工具，0.6B 小模型才能撑起完整管线。

> 类比：Android 手机厂商都基于 Google 开源的 AOSP 做自己的 ROM——小米的 MIUI 不是"从零研发的操作系统"，但 MIUI 里大量的定制工作是真功夫。HaS 之于 Qwen3-0.6B-Base 同理。

**为什么选 Qwen3-0.6B-Base 当基座？**（而不是自己预训练一个）

- **成本**：预训练要百万美元级投入，微调几乎免费；
- **能力继承**：Qwen3 预训练覆盖了中英法德西葡日韩——正好是 HaS 宣称支持的 8 种语言，直接继承；
- **尺寸**：0.6B 是端侧（手机/笔记本）可流畅运行的最大甜点位；
- **生态**：Qwen3 架构被 llama.cpp、transformers 等所有主流工具支持。

我们现在做的事，和玄武实验室做的事**本质相同，只是链条上又多了一环**：

```
阿里: 互联网文本 → [预训练] → Qwen3-0.6B-Base
玄武: 隐私标注数据 → [SFT] → HaS 模型
你:   CURP 数据  → [LoRA SFT] → 你的增强版 HaS
```

## 2.2 MIT 协议：为什么我们可以合法地二次训练

**开源协议**是作者给使用者的法律授权书。MIT 是最宽松的一种，全部义务就两条：保留版权声明、作者不担责。除此之外：

- ✅ 可以商用
- ✅ 可以修改（**包括二次训练**）
- ✅ 可以再分发（包括你微调出的新模型）

HaS 模型卡片上 `License: mit`，所以我们下载、微调、转格式、部署，全部合法。Qwen3-0.6B-Base 本身是 Apache 2.0 协议，同样宽松。

（对比：如果模型是 CC-BY-NC 这类"非商用"协议，商用微调就违法了——所以做二次开发前看 License 是必要动作。）

## 2.3 量化：为什么你的 GGUF 文件不能拿来训练

先看参数存储这件事。6 亿个数字，**每个数字用多少字节存，直接决定文件大小**：

| 格式 | 每个参数占用 | 0.6B 模型大小 | 特点 |
|---|---|---|---|
| fp32 | 4 字节 | ~2.4 GB | 训练黄金标准，精度最高 |
| **bf16** | 2 字节 | ~1.2 GB | 官方全量权重的格式，训练够用 |
| **Q8_0** | ~1 字节 | ~640 MB | **量化格式**，你本地 GGUF 的格式 |

**量化（quantization）是什么？** 把高精度的浮点数"压缩"成低精度整数的过程。

一个直观的例子。假设某层有 4 个 bf16 权重：

```
原始 (bf16):   0.0173,  -0.2541,  0.8912,  -1.0037
                 │ 找到这一组的最大值 1.0037，按比例映射到 [-128, 127] 的整数
                 ▼
量化后 (int8): 2,       -32,      113,     -128      + 一个缩放因子 0.00785
还原时:        2×0.00785=0.0157,  -32×0.00785=-0.2512, ... 
                 ▲ 注意：0.0173 变成了 0.0157——有信息损失，不可逆！
```

Q8_0 就是"每 32 个权重一组，各存成一个 8bit 整数 + 共用一个缩放因子"。体积缩小一半，推理时还更快（整数运算快），代价是**每个数字都损失了一点精度**。

**为什么推理可以容忍、训练不能容忍？**

- **推理**：只算一次前向传播，6 亿次微小误差大部分互相抵消，最终选出的 token 通常不变——"大概对"就够了；
- **训练**：要用梯度对参数做成千上万次**微小修正**（学习率 1e-4 级别）。可是量化后参数最小变化单位是 0.00785——梯度想做的 0.0001 级修正**直接低于量化精度，被舍入没了**。而且量化过程本身不可微（整数取整没有梯度），反向传播的数学链条直接断掉。

> 类比：全量权重（bf16 safetensors）是 **RAW 原图**，GGUF/Q8_0 是 **压缩后的 JPG**。你可以拿 JPG 看图（推理），但不能在 JPG 上做精细修图（训练）——每次保存都损失细节，而且很多编辑操作根本不支持。

**GGUF 又是什么？** GGUF 是 llama.cpp 项目定义的"打包格式"：把量化后的权重 + config + 分词器 + chat 模板**塞进一个文件**，专为本地推理优化。它是"终点站"格式，不是"工作区"格式。

**所以完整路径必然是（也是玄武实验室发布两个版本的原因）：**

```
官方全量权重 (safetensors, bf16)  ← 训练/微调用这个
      │  ① LoRA 微调（PyTorch，有梯度、有优化器）
      ▼
合并后的 HF 模型 (safetensors, bf16)
      │  ② llama.cpp 的 convert_hf_to_gguf.py（格式转换，不损失精度）
      ▼
GGUF (bf16, ~1.2GB)
      │  ③ llama-quantize → Q8_0（量化，损失一点精度换一半体积）
      ▼
GGUF (Q8_0, ~640MB)  ← 部署推理用这个，与官方发布格式一致
```

注意顺序：**先完成全部训练，最后才量化**。量化永远是最后一步。

## 2.4 llama.cpp、llama-server、GGUF 的分工

- **llama.cpp**：一个 C++ 写的本地推理引擎（GitHub 开源项目），专治"在消费级硬件上跑大模型"。
- **GGUF**：llama.cpp 的模型文件格式（上面讲过）。
- **llama-server**：llama.cpp 自带的 HTTP 服务程序，把 GGUF 模型包装成 OpenAI 兼容 API（`/v1/chat/completions`）。你启动参数里的 `-ngl 999`（所有层放 GPU）、`-c 8192`（上下文 8192 token）、`-ctk/-ctv q8_0`（KV cache 也量化省内存）都是推理优化开关。

我们训练用 **PyTorch**（因为它会算梯度），部署用 **llama.cpp**（因为它在 Mac 上推理快、省内存）——各干各最擅长的事，中间靠格式转换衔接。

# 第三章：训练原理深入——SFT、LoRA、格式一致性

## 3.1 HaS 的 NER 任务本质：条件文本生成

实测确认（`scripts/has_format.py`），HaS 的实体识别就是一个固定格式的问答：

```
输入（user 消息）:
  Recognize the following entity types in the text.
  Specified types:["person name","curp"]
  <text>Registro de usuario: nombre JUAN PEREZ, CURP HEMJ850312HNLRRS09</text>

输出（assistant 消息，纯 JSON）:
  {"person name":["JUAN PEREZ"],"curp":["HEMJ850312HNLRRS09"]}
```

模型并没有一个"NER 专用开关"——它就是在做老本行"逐 token 续写"，只是玄武实验室用 SFT 教会了它：**看到这种格式的输入，就续写这种格式的 JSON**。

**所以我们要"训练模型认识 CURP"，准确的表述是**：构造几千对 `(标准格式输入, 期望 JSON 输出)`，用梯度下降微调参数，让模型见到 CURP 相关的各种问法时，都能续写出正确的 JSON。

## 3.2 监督微调（SFT）：老师逐字带着写

SFT 的具体训练方式叫 **teacher forcing（教师强制）**：

```
拼成完整文本:
  <|im_start|>user
  Recognize the following entity types in the text.
  Specified types:["curp"]
  <text>...CURP HEMJ850312HNLRRS09...</text><|im_end|>
  <|im_start|>assistant
  {"curp":["HEMJ850312HNLRRS09"]}<|im_end|>

逐 token 计算损失:
  每个位置: 给左边所有 token, 模型预测下一个 token, 与标准答案算交叉熵
  例如: 模型看到 ...assistant\n{"curp":["HEMJ 时, 应该预测出 85 这个 token
```

一轮训练 = 几千条这样的文本，每条逐位置算 loss → 平均 → 反向传播 → 优化器改参数。

### Completion-only loss：只学答案，不学提问

一个关键技巧：把 prompt 部分（user 消息）的 token 标记为"**不算 loss**"（PyTorch 里 label 设为 -100），**只有 assistant 回答部分的 token 参与损失计算**。

为什么？

1. **省容量**：我们不需要模型学会"怎么提问"，提问永远由用户/程序提供；
2. **防抢话**：如果连 prompt 一起学，模型可能学会"回答完再自己编一个新 `<text>` 继续自问自答"，推理时输出就脏了。

（实现见 `scripts/train_lora.py` 的 `HasSFTDataset`，就十几行代码。）

## 3.3 LoRA：不动 6 亿参数，只加"小补丁"

1.4 节说过全参数训练的问题：优化器状态就要 ~5GB 内存，而且 6 亿参数一起动，很容易把玄武实验室训好的能力"冲掉"（**灾难性遗忘**）。

**LoRA（Low-Rank Adaptation，低秩适配）**的思路非常聪明：

```
原层计算:  y = Wx            (W 是 1024×1024 = 104万 参数, 冻结不动)
LoRA 后:   y = Wx + BAx      (A: 16×1024, B: 1024×16, 只有 3.2万 参数, 只训练它俩)
```

数学直觉：W 已经是"好模型"，我们需要的修改量 ΔW 通常不需要完整的 1024 个自由度——微调带来的改变是"低秩的"（low-rank），可以用两个小矩阵的乘积 B·A（秩 r=16）近似表达。**r（秩）就是 LoRA 唯一的尺寸旋钮**：r=16 时参数量是原矩阵的 3%，却足以表达"新增 CURP 识别规则"这种程度的修改。

**LoRA 的三大好处，在我们的场景全部命中：**

| 好处 | 在本项目的体现 |
|---|---|
| 省内存 | 可训练参数 ~300 万（0.5%），优化器状态 ~25MB，24GB Mac 轻松跑 |
| 防遗忘 | 原 6 亿参数**冻结**，最坏情况也只是"补丁没打好"，不会把原模型搞坏 |
| 零推理成本 | 训练完把 B·A **加回** W（`merge_and_unload`），得到的还是普通模型，转 GGUF 后推理速度和原来一模一样 |

## 3.4 格式一致性：微调失败的第一大原因

模型对格式的敏感超出直觉。训练数据的 prompt 与线上服务的 prompt 哪怕差一个空格，效果都会打折——因为模型学到的是"**这个精确文本模式** → 这个 JSON"，不是抽象规则。

本项目死守三条一致性：

1. **指令文本一致**：`Recognize the following entity types in the text.\nSpecified types:[...]\n<text>...</text>`——来自官方示例逆向 + 本机 llama-server 实测验证（`scripts/has_format.py`）；
2. **chat 模板一致**：`<|im_start|>user ... <|im_end|> <|im_start|>assistant ...` 这套包装来自模型仓库的 `chat_template.jinja`。训练时 `tokenizer.apply_chat_template` 用它渲染；转 GGUF 时这个模板**自动打包进模型文件**，llama-server 加载后用的还是它——同一份文件，天然一致；
3. **输出格式一致**：JSON 无空格分隔、key 顺序 = 请求顺序、未命中 = `[]`。

## 3.5 数据设计：教"概念"，而不是教"句子"

如果只喂 `"CURP xxx" → 识别` 一种句式，模型会**记住句子**。`scripts/gen_dataset.py` 从四个维度制造多样性：

1. **类型别名**：`curp`/`CURP`/`mx_curp`/`Mexican CURP`/`id number`/`ID number`/`mx_id`… 都映射到 CURP。这是基线模型最弱的点（实测别名一致性仅 20%），也是训练收益最大的点；
2. **上下文**：西/英/中 × 20+ 句式模板；CURP 值按**官方结构规则随机合成**（18 位：姓名首字母+生日+性别+州码+同名单元+校验位，校验位算法已用真实 CURP 验证），几千条样本几乎不重复——模型没法靠背；
3. **负样本**：正文没有 CURP 却请求 `curp` → `{"curp":[]}`；正文是 INE 选民证号（也是 18 位！）却请求 `curp` → 仍为空。教模型"**不是什么**"和教"是什么"同样重要（基线模型在这里精确率只有 32%）；
4. **防遗忘混合**：~30% 样本是模型原有能力（人名/电话/邮箱/IP/MAC），训练时不丢老本事。

## 3.6 评估：怎么知道训练成功了？

`scripts/evaluate.py` 用**训练中没出现过的句式模板**生成评估集（测泛化，不是测背诵），指标：

- **JSON 合法率**：输出格式有没有被训坏；
- **完全匹配率**：整个 JSON 与期望一字不差的比例；
- **实体级 P/R/F1**：精确率（找出来的有多少是对的）、召回率（该找的有多少找到了）、F1（两者调和平均）；
- **分场景**：`curp`（新能力）/ `generic`（旧能力回归）/ `distractor`（反例鉴别）；
- **别名一致性**：同一段文本分别用 `curp`/`id number`/`mx_curp` 问，三次答案是否一致且正确。

**标准做法：微调前后各跑一次，对比报告。** 判断标准：

- `curp` 场景 F1 显著上升 → 新能力学到了；
- `generic` 场景不下降（>2 个点算事故）→ 没有遗忘；
- `distractor` 精确率上升 → 学会了边界。

# 第五章：超参数怎么调

| 参数 | 默认 | 含义与调节 |
|---|---|---|
| `epochs` | 3 | 全部数据过几遍。训练 loss 还在降可加；评估集反而变差=过拟合，减 |
| `lr`（学习率） | 1e-4 | 梯度下降的步长（见 1.4）。LoRA 常用 5e-5~2e-4。遗忘明显→调低；学不动→调高 |
| `lora_r` | 16 | LoRA 补丁的秩（见 3.3）。格式单一任务 16 够；教很多新类型可到 32~64 |
| `batch×accum` | 4×4 | 有效 batch=16：每算 16 条样本的梯度才更新一次参数，更稳 |
| 训练集规模 | 2000 | 格式规整的任务 1k~5k 足够，**多样性比数量重要** |

# 第六章：常见问题

1. **输出不是 JSON / 格式错乱** → 训练数据格式没对齐（3.4 节），打印一条训练样本的完整渲染文本检查。
2. **CURP 会了，人名变差** → 灾难性遗忘。调低 lr、减 epochs、提高 generic 样本比例。
3. **只认训练里的句式** → 模板多样性不够，扩 `TRAIN_TEMPLATES`。
4. **把 INE 也标成 CURP** → distractor 负样本不够，提高比例。
5. **转 GGUF 报错** → 确认合并目录有 `config.json`/`tokenizer.json`/`chat_template.jinja`；llama.cpp 用最新 master。

# 第七章：进阶方向

- **训练 Hide 任务**：hide 的 prompt 格式与 NER 不同（输出带三级语义标签），先逆向模板（可用 OpenClaw 的 has CLI 抓包），再用同样方法造数据；
- **教更多证件**：RFC（墨西哥税号）、INE、护照…在 `gen_dataset.py` 加一个生成函数 + 别名表即可；
- **更大规模**：`--train-size` 提到 5000+，或混入真实语料（注意脱敏合规）；
- **全参数训练**：数据 10 万级且要教全新行为模式时再考虑，需要更强算力（复习 1.4 节的内存账）。
