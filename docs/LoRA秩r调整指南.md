# LoRA 秩（r）调整指南

> 适用场景：本项目（HaS NER 微调）及类似的 LoRA SFT 任务。
> 核心结论：**r 由"模型需要新学会的信息量"决定，而不是由"类型数量"直接决定。**

## 1. 直觉：类型数量 ≠ 需要的容量

```
任务 A：教 100 个新证件类型，但都是"认模式 → 按固定 JSON 输出"
任务 B：教 3 个类型，但每个要学复杂的上下文判断规则
```

任务 A 需要的容量可能比任务 B **还少**——100 个类型共享同一套技能（认模式、
输出 JSON），模型只需为每个类型存一个很小的"模式指纹"。

本项目实证：1 个新类型（CURP）+ 别名 + 上下文判断，r=16 已 F1=0.996，
容量还有富余（r=8 可能都够）。

## 2. 实用对照表

| 场景 | 建议 r | alpha | 依据 |
|---|---|---|---|
| 1~10 个新类型，格式/模式类学习 | **8~16** | 2×r | 技能已会，只需存"模式指纹" |
| 10~50 个新类型，或单类型但规则复杂（多语言、上下文歧义多） | **32** | 64 | 类型间差异需要更多独立方向表达 |
| 50+ 个类型，或要教"知识"而不只是"格式" | **64~128** | 2×r | 接近 LoRA 能力上限 |
| r=128 还学不动 | 停止加 r | — | 该换全量微调或更多数据，LoRA 天花板到了 |

## 3. 比"类型数量"更可靠的判断信号

不要预设 r，**让训练结果告诉你**。

### 该提高 r 的信号（欠拟合）

- train loss 降不下去（本项目可降到 ~0.0001；若卡在高位说明容量不足）
- **训练集**上的完全匹配率都上不去——连教过的都学不会
- 多种新类型之间互相"串味"（A 类型报成 B 类型）

### r 太高的信号（过拟合）

- train loss → 0 但 eval loss 反而上升
- 防遗忘场景（generic）指标下降——旁路太大开始干扰原模型
- 对训练模板措辞过拟合，换个句式就失效

## 4. 调整时的配套规则

1. **alpha 跟着 r 走**：保持 `alpha = 2×r`。alpha/r 是更新幅度的缩放系数，
   只加 r 不加 alpha 等于白加。
2. **数据量要跟上**：r 翻倍 ≈ 可训练参数翻倍。经验值：每个新类型至少
   几百条样本，再考虑升 r。

## 5. 本项目的扩展建议

| 未来扩展 | 建议 |
|---|---|
| 加 1~2 个证件类型（RFC、INE 细化） | 保持 r=16，改 `gen_dataset.py` 加类型即可 |
| 一次性加 10+ 个拉美证件类型 | r=32, alpha=64，总数据量加到 5000+ |
| 大量"形似但规则不同"的类型 | 先 r=32 看 train loss，降不动再 r=64 |

## 6. 一句话原则

> **r=16 起步，看 train loss 和训练集匹配率，学不会才加倍**——
> 而不是按类型数量预先放大。
